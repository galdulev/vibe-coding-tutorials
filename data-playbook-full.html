pre-flight: context preparation

Goal: Set up AI with everything it needs to succeed

step 1: create your context file

File: .cursorrules or CONTEXT_FOR_AI.md in project root

# Project Context for AI

## Database Schema
[Include table definitions with annotations]

## Business Context
[Field meanings, KPIs, business rules]

## Data Quality Notes
[Known issues, refresh schedules, gotchas]

## Common Queries
[Examples of typical analysis patterns]

## User Personas & Use Cases
[Who uses this, what they need]

Time Investment: 2-4 hours initially | Saves 10+ hours in iteration

step 2: gather sample data

Create small CSV files or SQL dumps with:

  • Representative data (5-100 rows)
  • Edge cases (nulls, outliers, special values)
  • Different time periods

step 3: define your analysis goal

Template:

GOAL: [What decision or insight are we enabling?]
AUDIENCE: [Who will use this?]
KEY METRICS: [Top 3-5 metrics that matter]
TIME FRAME: [Historical period, real-time, forecast?]
OUTPUT: [Report, dashboard, API, scheduled email?]

phase 1: exploratory analysis

Goal: Understand your data and validate assumptions

step 1.1: initial data exploration

Prompt Template:

I need to analyze [DATASET/TABLE]. Please help me:

1. Show me the data structure (columns, types, sample values)
2. Check for data quality issues (nulls, duplicates, outliers)
3. Generate summary statistics for key fields
4. Identify relationships between tables

Context:
- [Brief business context]
- [Any known data issues]
- [What I'm trying to learn]

Expected Output:

  • Python script with pandas or SQL queries
  • Summary statistics
  • Data quality report
  • Recommendations for cleaning

step 1.2: ask specific analysis questions

Prompt Pattern:

Using [DATASET], answer: "[BUSINESS QUESTION]"

Please:
1. Write the query/analysis code
2. Explain your approach
3. Show results with visualization
4. Flag any caveats or limitations

Example data: [paste 3-5 rows]

step 1.3: iterate and refine

[If results look wrong]
"The results show [UNEXPECTED PATTERN]. This doesn't match business expectations. 
Can you:
1. Verify the join conditions
2. Check for data filtering issues
3. Show me the intermediate steps"

[If you need different angle]
"Good start. Now let's also analyze [ADDITIONAL DIMENSION]. 
Can you modify the query to also show [NEW METRIC]?"

phase 2: building analytics

Goal: Create reusable analysis code and metrics

step 2.1: define metrics clearly

Prompt Template:

I need to create a metric calculation for: [METRIC NAME]

Business Definition: [How business defines it]
Formula: [Math formula if available]
Data Sources: [Tables/fields involved]
Filters: [What to include/exclude]
Grouping: [Time period, region, category?]
Edge Cases: [Special handling needed]

Please create a Python function or SQL query that:
1. Implements this metric correctly
2. Handles edge cases
3. Is reusable and parameterized
4. Includes validation checks

step 2.2: build analysis pipeline

Prompt Template:

Create an analysis pipeline that:

INPUT: [Data source, time range, filters]
STEPS:
  1. [Load/extract data]
  2. [Clean/transform]
  3. [Calculate metrics]
  4. [Aggregate/summarize]
  5. [Export results]
OUTPUT: [Format, destination]

Requirements:
- Modular code (separate functions for each step)
- Error handling
- Logging
- Can be scheduled/automated

step 2.3: validate results

Prompt:

Help me validate this analysis:

1. Show sample calculations step-by-step for 2-3 records
2. Compare results to manual calculation
3. Check for common errors:
   - Off-by-one errors in date ranges
   - Double-counting in joins
   - Missing data handling
   - Timezone issues
4. Suggest test cases I should verify

best practices

DO's

  1. Start Small, Iterate
    • Begin with simple queries/charts
    • Add complexity gradually
    • Test at each step
  2. Be Specific in Prompts 
    BAD: "Create a dashboard"

GOOD: "Create a Streamlit dashboard showing 3 KPI cards and a line chart of 
       monthly trends, with a date range filter"
  3. Provide Context
    • Share sample data
    • Explain business logic
    • Mention edge cases
  4. Validate Results
    • Always spot-check calculations
    • Compare to known benchmarks
    • Test with edge case data
  5. Build Reusable Components
    • Ask for modular functions
    • Request parameterized queries
    • Create config files for settings

DON'Ts

  1. Don't Assume AI Knows Your Data
    • Even with context files, be explicit
    • Provide examples in prompts
  2. Don't Skip Validation
    • AI can make plausible-looking mistakes
    • Always verify critical metrics
  3. Don't Overcomplicate Prompts
    • One major task per prompt
    • Break complex requests into steps
  4. Don't Ignore Errors
    • If output looks wrong, investigate
    • Ask AI to explain its logic
  5. Don't Forget Edge Cases
    • Test with nulls, zeros, extremes
    • Consider timezone issues

common pitfalls & solutions

pitfall 1: wrong aggregations

Symptom: Numbers don't match manual calculations

Solution:

Prompt: "Show me the step-by-step calculation for customer_id='C123', 
breaking down each aggregation and join. Then verify the logic."

pitfall 2: time zone confusion

Symptom: Date-based metrics are off by a day

Solution:

Prompt: "All timestamps in my database are UTC. When calculating daily 
metrics, convert to [TIMEZONE] first, then aggregate by date. 
Show me how you're handling timezones in the code."

pitfall 3: cartesian explosion in joins

Symptom: Counts are way too high

Solution:

Prompt: "The counts look inflated. Please:
1. Show the join logic explicitly
2. Check for many-to-many relationships
3. Add DISTINCT where appropriate
4. Show row counts at each join step"

pitfall 4: hardcoded values

Symptom: Dashboard breaks when data changes

Solution:

Prompt: "Refactor this code to use configuration:
- Move all hardcoded filters to config.yaml
- Parameterize date ranges
- Make chart colors configurable
- Extract SQL queries to separate files"

pitfall 5: poor error handling

Symptom: Dashboard crashes on bad data

Solution:

Prompt: "Add comprehensive error handling:
1. Wrap data loading in try-except
2. Validate data shape/types before processing
3. Show user-friendly error messages
4. Log errors for debugging
5. Provide fallback behavior (show message vs crash)"

example workflow: end-to-end

scenario: create a sales performance dashboard

Step 1: Initial Prompt

I need to create a sales performance dashboard for regional managers.

Data: PostgreSQL database with tables 'sales', 'products', 'regions'
Goal: Show monthly sales trends, top products, regional comparison
Audience: Regional sales managers (non-technical)
Framework: Streamlit (already installed)

First, help me:
1. Understand the data structure
2. Define key metrics
3. Design the layout

Step 2: Define Metrics

Based on discussion, create Python functions for:

1. total_sales(start_date, end_date, region=None)
   - Returns: Total revenue, number of transactions, avg transaction value
   
2. sales_trend(start_date, end_date, grouping='month')
   - Returns: DataFrame with dates and sales values
   
3. top_products(start_date, end_date, top_n=10)
   - Returns: DataFrame with product names and sales, sorted

Step 3: Build Dashboard Shell

Create Streamlit dashboard with:

1. Sidebar: Date range picker, region selector
2. Main area:
   - Row 1: 3 KPI cards (total sales, transactions, avg value)
   - Row 2: Line chart (sales trend)
   - Row 3: Two columns
     - Left: Bar chart (top 10 products)
     - Right: Table (regional breakdown)

Use st.columns for layout. Use dummy data for now.

Step 4: Connect Real Data

Connect to database using these credentials: [REDACTED]

Replace dummy data in the dashboard with real queries.
Add caching for performance using @st.cache_data.

Step 5: Polish

Improvements needed:
1. Add loading spinners
2. Format numbers (currency, percentages)
3. Add title and description
4. Make charts interactive (click to filter)
5. Add export to Excel button
6. Improve color scheme (use company colors: #FF6B6B, #4ECDC4)

Step 6: Deploy

Help me deploy this dashboard:
1. Create requirements.txt
2. Write deployment instructions
3. Create README.md
4. Add environment variable handling for DB credentials

success checklist

Before considering your dashboard complete:

  • All metrics calculated correctly (spot-checked)
  • Edge cases handled (nulls, zeros, extremes)
  • Performance acceptable (<3s load time)
  • Responsive on different screen sizes
  • Error messages clear and actionable
  • Data refresh working properly
  • Filters and interactions work as expected
  • Documentation complete (user guide + technical)
  • Code reviewed and refactored
  • Tested by actual users
  • Deployment instructions clear
